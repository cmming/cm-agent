# 指定提交文件导出设计

## 目标

提供一个 PowerShell 脚本，将指定 Git 提交相对首个父提交新增、修改和重命名后的文件，按仓库相对目录结构导出到目标目录，供目标工作区直接覆盖。

## 方案选择

可选方案包括 PowerShell、Shell 脚本和 Java 命令行工具。项目的日常开发说明使用 PowerShell，且该任务面向当前 Windows 工作区，因此选择 PowerShell；它无需新增构建依赖，也可直接调用 Git。

## 接口

脚本路径为 `scripts/export-commit-files.ps1`，参数如下：

```powershell
.\scripts\export-commit-files.ps1 -Commit <提交哈希> -OutputDirectory <导出目录>
```

`Commit` 必须解析为一个存在的提交对象。`OutputDirectory` 必须不存在或为空目录，以防旧导出文件和本次结果混合。

## 导出流程

1. 校验当前目录位于 Git 工作区，并解析提交及其首个父提交。
2. 使用 `git diff-tree` 获取首个父提交到目标提交之间的名称状态。
3. 对 `A`、`M` 和 `R` 状态，取状态记录中的新路径；从目标提交读取该路径内容，创建目标目录中的父目录后写入文件。
4. 对 `D` 状态，将旧路径写入 `deleted-files.txt`，不复制文件。
5. 将实际导出的相对路径写入 `exported-files.txt`。两个清单均采用 UTF-8 编码。

## 边界与错误处理

- 不处理合并提交的所有父边；固定以首个父提交作为比较基线，符合 `git show` 的常见语义。
- 根提交没有父提交时明确失败，避免隐式选择空树造成调用者误解。
- 子模块、类型变更或无法从目标提交读取的路径明确失败，避免产生不完整覆盖包。
- 目录、可执行位和符号链接元数据不作为交付范围；导出目标是普通可覆盖文件。

## 验证

通过临时 Git 仓库的 PowerShell 自动化测试验证：修改文件仍保留嵌套路径、重命名导出新路径、删除路径进入清单、无效提交被拒绝。README 提供中文用法和覆盖前需查看删除清单的说明。

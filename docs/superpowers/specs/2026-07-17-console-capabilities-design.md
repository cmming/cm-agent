# CM Agent 可操作管理控制台设计

## 1. 背景与目标

CM Agent 已提供登录、当前用户、Agent 管理、工具治理、Agent 运行、运行历史、运行详情和审计查询等后端接口，但现有轻量控制台只覆盖创建与调试主路径，无法完整展示已经交付的接口能力。

本次改造面向平台使用者，将现有控制台升级为可操作管理控制台。控制台必须基于当前真实接口形成完整操作闭环，不新增统计、编辑、删除、取消运行等后端能力，也不展示尚未实现的功能。

## 2. 范围

### 2.1 包含范围

- 独立登录页，以及登录后的当前用户信息。
- Agent 列表、详情和创建。
- Tool 列表、创建和向指定 Agent 授权。
- 执行 Agent、查看运行历史和运行详情。
- 查看运行关联的工具调用。
- 使用后端游标加载更多运行记录和审计日志。
- 基于已加载真实数据生成能力总览。
- 提供健康检查与 OpenAPI 快捷入口。
- 桌面端和窄屏响应式布局、键盘焦点和可读状态提示。

### 2.2 不包含范围

- 不新增或修改后端业务接口。
- 不实现 Agent、Tool 的编辑或删除。
- 不实现手动取消、流式输出、多轮会话或 HITL。
- 不新增后端统计或聚合接口。
- 不引入 React、Vue、模板引擎或 Node 前端构建链。
- 不持久化 JWT、用户名或密码。

## 3. 技术方案

继续使用 `cm-agent-console` 中随 JAR 发布的原生 HTML、CSS 和 JavaScript：

- `index.html` 提供登录视图、控制台外壳、导航和各模块语义结构。
- `assets/app.js` 管理认证状态、接口请求、页面切换、数据刷新、表单提交和安全渲染。
- `assets/styles.css` 提供视觉系统、布局、状态样式和响应式规则。

页面保持单入口单页结构，通过前端状态切换模块，不增加浏览器路由或服务端页面 Controller。JavaScript 仍以单文件交付，但函数按状态管理、请求、渲染、业务动作和格式化职责组织，避免把逻辑直接堆叠在事件监听器中。

## 4. 信息架构

### 4.1 未认证状态

未登录时只显示独立登录页，包括产品标识、用户名、密码、提交状态和错误提示。登录成功后使用返回的 JWT 调用 `/api/auth/me` 校验当前用户，再进入控制台。

JWT 只保存在页面内存中。刷新页面、关闭标签页或收到 `401` 后必须重新登录。密码不得回显、记录或写入持久存储。

### 4.2 已认证状态

控制台采用深色左侧导航和浅色主内容区：

- **能力总览**：展示当前用户、Agent 数、Tool 数、最近运行和快捷操作。
- **Agent 管理**：展示 Agent 列表、详情和创建表单，并提供运行调试与运行历史入口。
- **工具治理**：展示 Tool 列表、创建表单和向 Agent 授权表单。
- **运行记录**：选择 Agent 后加载运行列表，支持加载更多和查看单次运行详情及工具调用。
- **审计日志**：展示审计事件并使用游标加载更多。
- **辅助入口**：以外链方式提供 OpenAPI 和健康检查。

侧边栏只负责模块导航。详情内容在主区域内通过面板呈现，不新增弹窗框架或独立路由。

## 5. 数据流

### 5.1 登录与初始化

1. 用户提交用户名和密码至 `POST /api/auth/login`。
2. 页面在内存中保存返回的访问令牌。
3. 页面调用 `GET /api/auth/me` 校验令牌并加载当前用户。
4. 页面并行调用 `GET /api/agents` 和 `GET /api/tools`。
5. 若存在 Agent，默认选择第一个 Agent，并调用 `GET /api/agents/{agentId}/runs?limit=20` 加载最近运行。
6. 总览根据当前已加载的 Agent、Tool 和运行数据计算展示内容；不得把局部数据表述为全局统计。

### 5.2 数据变更后的刷新

- 创建 Agent 成功后刷新 Agent 列表，并选择新建 Agent。
- 创建 Tool 成功后刷新 Tool 列表。
- Tool 授权成功后保留当前选择并显示成功反馈；现有接口不提供授权列表，因此页面不推断或展示完整授权关系。
- Agent 运行完成后刷新当前 Agent 的运行历史，并允许打开本次运行详情。
- 各操作只刷新受影响的数据区域，避免无关请求和页面闪烁。

### 5.3 游标分页

运行历史和审计日志直接使用响应中的 `nextCursor`：

- 有游标时显示“加载更多”。
- 无游标时显示“已加载全部”。
- 新查询或切换 Agent 时清空旧游标和旧列表。
- 不根据游标伪造总页数或页码。

## 6. 接口映射

| 控制台能力 | 后端接口 |
| --- | --- |
| 登录 | `POST /api/auth/login` |
| 当前用户 | `GET /api/auth/me` |
| Agent 列表 | `GET /api/agents` |
| Agent 详情 | `GET /api/agents/{id}` |
| 创建 Agent | `POST /api/agents` |
| Tool 列表 | `GET /api/tools` |
| 创建 Tool | `POST /api/tools` |
| Tool 授权 | `POST /api/tools/{id}/grants` |
| 执行 Agent | `POST /api/agents/{agentId}/runs` |
| 运行历史 | `GET /api/agents/{agentId}/runs` |
| 运行详情与工具调用 | `GET /api/agents/{agentId}/runs/{runId}` |
| 审计日志 | `GET /api/audit-events` |
| 健康检查 | `GET /actuator/health` |
| OpenAPI | `/swagger-ui/index.html` |

## 7. 交互与状态

- 提交过程中禁用对应按钮，并显示进行中状态，防止重复提交。
- 列表分别提供加载中、空数据、加载失败和正常数据状态。
- 选中 Agent、Tool 或 Run 时提供明确的选中样式和详情标题。
- 后端没有对应能力的操作不得显示为禁用按钮，避免误导用户。
- 退出登录时清空令牌、用户信息、列表、选中项、游标和敏感表单值。
- 页面不得通过 `innerHTML` 注入接口返回内容；动态文本使用 `textContent` 或显式创建节点。

## 8. 错误处理与安全

通用请求层统一解析 JSON 和文本错误，并根据状态处理：

- `400`：显示可读的参数或表单错误。
- `401`：清空认证状态并返回登录页。
- `403`：提示权限不足，不重试或绕过授权。
- `404`：提示资源不存在，并刷新可能过期的列表。
- `5xx`：显示服务暂不可用，不暴露堆栈、SQL 或内部配置。

页面和错误提示不得展示 JWT、密码、模型 API Key、数据库凭据、完整 JDBC URL 或底层 SQL。服务端返回的所有动态文本均作为纯文本渲染。

## 9. 视觉与响应式设计

- 深蓝色侧栏用于稳定导航，主内容区使用浅灰背景和白色卡片。
- 蓝色作为主操作色，绿色、橙色、红色分别辅助表达成功、进行中和失败，但必须同时显示文字。
- 桌面端固定侧栏；窄屏下折叠为顶部菜单，卡片改为单列，宽表格改为可横向滚动或纵向信息块。
- 所有输入拥有可见标签，交互控件具备明显键盘焦点。
- 状态与错误区域使用 `aria-live`，导航和主要区域使用语义化元素。
- 页面不依赖远程字体、图标库或第三方静态资源。

## 10. 测试与验收

实现遵循测试先行：

1. 扩展控制台静态资源烟测，先断言独立登录页、侧边导航和核心能力入口，使测试因新结构缺失而失败。
2. 实现最小页面结构与样式使烟测通过。
3. 为可独立执行的 JavaScript 逻辑增加测试或脚本检查，覆盖错误格式化、`401` 认证失效、游标追加和动态文本安全渲染。
4. 补充或调整 MockMvc 测试，确认首页和静态资源可访问，既有 API 安全规则不被改变。
5. 使用 JDK 21 和 Maven 3.9+ 运行服务端相关测试及跳过测试打包。

验收标准：

- 使用现有 bootstrap admin 能完成登录并进入控制台。
- 页面可操作本设计接口映射中的全部业务接口。
- Agent、Tool、运行和审计的空状态、成功状态及错误状态均有可读反馈。
- 运行和审计游标可正确追加数据，不重复覆盖已有记录。
- `401` 会话失效后返回登录页，退出后不残留令牌和密码。
- 页面不展示未实现能力，也不新增后端业务接口或前端依赖。
- 桌面端与窄屏均可完成主要操作，键盘焦点和状态文本清晰可见。

## 11. 文档影响

实现完成后更新 `README.md` 或相关生产文档，说明控制台已覆盖的能力、访问入口和当前限制；同步更新 `docs/release-notes.md` 记录用户可见行为变化。
